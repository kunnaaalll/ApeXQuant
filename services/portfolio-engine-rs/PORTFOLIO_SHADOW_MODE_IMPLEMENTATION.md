# APEX V3 PORTFOLIO ENGINE: SHADOW MODE IMPLEMENTATION

## Philosophy
**Observe. Compare. Measure. Report. Never interfere.**

The Shadow Mode is designed to run the Rust Portfolio Engine in parallel with the existing TypeScript implementation. It receives the same inputs as the production TypeScript engine, processes them, and then compares its generated outputs and state against the actual outputs generated by the TypeScript engine.

Crucially, **no live decisions depend on the Portfolio Engine yet**. The Shadow Engine cannot mutate live orders, positions, or portfolio state. It is strictly an observer.

## Components

### 1. Comparison Engine (`src/shadow/comparison.rs`)
The `PortfolioComparison` struct provides the core logic to diff two JSON structures (or domain objects) representing the state, exposure, heat, allocation, and recommendations. 

Matches are classified as:
- `ExactMatch`: 100% parity down to the exact float representation.
- `CloseMatch`: Functionally identical, but floating-point drift is within an acceptable epsilon (e.g., `< 0.0001%`).
- `MinorDifference`: Output differences that do not trigger different threshold behaviors (e.g., quality drop from 95.1 to 94.9).
- `MajorDifference`: Outputs that cross a behavioral threshold.
- `Mismatch`: Fundamentally opposed outputs (e.g., TS says Reduce, Rust says Increase).
- `MissingData`: One side produced output while the other did not.

Every `DifferenceDetail` tracks the field, expected value, actual value, and reason.

### 2. Drift Engine (`src/shadow/drift.rs`)
The `DriftMonitor` tracks the accumulation of `MinorDifference` and `CloseMatch` classifications over time. 
Drift is assessed across specific domains:
- Quality Drift
- Health Drift
- Drawdown Drift
- Heat Drift
- Recommendation Drift
- Analytics Drift

The state escalates from `Stable` -> `Watch` -> `Elevated` -> `Critical` based on predefined thresholds.

### 3. Statistics Engine (`src/shadow/statistics.rs`)
Aggregates the comparisons and drift assessments into rolling window metrics (1h, 1d, 1w, 1m).
It tracks:
- Agreement %
- Exact Match %
- Close Match %
- Major Mismatch %
- Average Drift
- Max Drift

### 4. Shadow Storage (`src/shadow/shadow_storage.rs`)
Provides an append-only, replayable persistence layer for shadow artifacts.
All `ShadowEvent` and `ShadowSnapshot` structs are stored here. It supports storing to SQLite or Postgres to allow offline analytics and replay testing.

### 5. Reporter (`src/shadow/reporter.rs`)
Generates standardized reports (`ShadowReport`) on a Daily, Weekly, and Monthly basis. These are output as Markdown or JSON for both human and machine-readable consumption.

### 6. Validator & Alerting (`src/shadow/validator.rs`)
The `PortfolioValidator` acts as the final gatekeeper for evaluating parity. It determines `Go` or `No-Go` based on:
- State agreement >99%
- Recommendation agreement >95%
- Health drift <5%
- Quality drift <5%
- Drawdown drift <2%
- Heat drift <2%
- Zero panics
- Zero divergence

It emits `ShadowAlert` events (Info, Warning, Critical) when drift accelerates or mismatches spike.

## Edge Cases & Known Limitations
- **Timestamp Drift:** Events ingested from the message bus might arrive with slight timestamp variances. Comparison logic must allow for `CloseMatch` based on epsilon time windows.
- **Float Rounding:** Rust and TS handle floating point arithmetic differently. The `comparison.rs` engine must define a strict epsilon for `CloseMatch`.
- **Order of Operations:** Map/Dict iterations might have different deterministic orders between TS and Rust. Array comparisons should be order-agnostic unless sequence implies priority.

## Failure Scenarios
If the Rust engine panics or crashes, the event must be caught and logged as a `ShadowEvent::CriticalFailure`. The Shadow Mode must never bring down the node it runs on.
